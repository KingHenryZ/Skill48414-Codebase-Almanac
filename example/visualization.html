Feature Map

User Registration

Complete sign-up flow with email validation, password hashing, and JWT issuance.

  • POST /api/auth/register creates a new user with email, password, and name
  • Passwords hashed with bcryptjs at cost factor 12 before storage
  • Email validation shared from `@spendwise/shared` ensures consistent format checks
  • Returns a signed JWT token (7-day expiry) alongside the user profile
User Login

Email and password authentication with secure hash comparison and token issuance.

  • POST /api/auth/login authenticates via email + password comparison
  • bcrypt.compare verifies the stored hash — timing-safe by design
  • Issues a fresh JWT with `userId` and `role: member` claims on success
  • Returns a generic 401 "Invalid credentials" for both wrong-email and wrong-password to prevent user enumeration
Profile Retrieval

Authenticated endpoint returning user details, protected by JWT middleware.

  • GET /api/auth/me returns the authenticated user's id, email, and name
  • Protected by the authenticate middleware — requires a valid Bearer token
  • Queries the users table by the JWT's `userId` claim
  • Returns 404 if the token is valid but the user record has been deleted
Shared Validation Library

Cross-package input validation for emails, categories, and expenses.

  • validateEmail — regex-based format check reused by both API and future clients
  • validateCategory — enforces name length (≤80 chars), hex color format, icon length (≤8 chars)
  • validateExpense — checks positive amount, valid date, description length (≤500 chars)
  • Exported from @spendwise/shared so any new package inherits the same rules

Product Architecture

  • The project is organized as an npm workspace monorepo with 2 packages under `packages/`.
  • @spendwise/api owns the Express server, all route definitions, middleware, the database layer, and the auth controller — it is the sole runtime package.
  • @spendwise/shared exports pure validation functions (`validateEmail`, `validateCategory`, `validateExpense`) consumed by the API via a workspace `file:` dependency.
  • The config/ directory holds a shared ESLint configuration applying `typescript-eslint` recommended rules across all packages.
  • The root `package.json` wires workspace-level scripts (`dev`, `build`, `lint`) that delegate to the appropriate package, keeping the developer workflow centralized.

Design Patterns

Static Service Class

AuthService uses static async methods for a clean, instantiation-free API.

  • AuthService exposes `register`, `login`, and `getProfile` as static async methods
  • Each method owns its own validation, database query, and response shaping — no separate repository layer
  • Error signaling uses `Object.assign(new Error(...), { status })` to carry HTTP status codes through the middleware chain
Layered Middleware Pipeline

Requests pass through security, logging, and rate-limiting layers before any route.

  • Requests pass through helmet → CORS → JSON parser → requestLogger → rateLimiter before reaching any route
  • The errorHandler sits at the end to catch thrown errors and normalize JSON responses
  • A catch-all 404 handler ensures unmatched routes return structured JSON, not Express defaults
Input Sanitization at Boundary

Emails are normalized and passwords validated before touching the database.

  • Emails are lowercased and trimmed inside `AuthService.register` before any DB query
  • Password minimum length is validated inline (8 chars) before hashing
  • The shared `validateEmail` regex rejects whitespace and missing `@` — applied before the controller ever touches the database

Backend Analysis

  • server.ts is the Express entry point — it imports `database.ts` as a side effect (creating the SQLite DB and running migrations on startup), then mounts middleware and routes.
  • The middleware pipeline executes in strict order: `helmet` (CSP + security headers) → `cors` (origin whitelist or wildcard) → `express.json` (10 KB body limit) → `requestLogger` (timing) → `rateLimiter` (100 req / 15 min per IP).
  • authRoutes is the only route group — mounted at `/api/auth` — and delegates to AuthService static methods for registration, login, and profile retrieval.
  • AuthService combines controller logic and data access in one class — `bcryptjs` handles password hashing (cost 12), `jsonwebtoken` issues 7-day tokens with `userId` + `role` claims.
  • errorHandler maps `err.status` to HTTP status codes and suppresses raw messages in production (500s → generic "Internal server error").
  • rateLimiter is a hand-rolled in-memory `Map<string, {count, resetAt}>` — simple but not cluster-safe and grows unbounded without eviction.
  • The shared package re-exports `validateEmail`, `validateCategory`, and `validateExpense` from a single barrel `index.ts`, consumed via the npm workspace `file:` link.
  • Build tooling uses tsx for dev (watch mode) and tsc for production — no bundler, no test runner configured yet.

Frontend Analysis

  • No front-end package exists yet — the monorepo reserves the `packages/*` workspace pattern for a future `@spendwise/web` client.

Code Quality Signals

Clean Middleware Separation

Each middleware lives in its own single-responsibility file — `authenticate.ts`, `rateLimiter.ts`, `requestLogger.ts`, `errorHandler.ts` — making the pipeline easy to reorder or extend.

Workspace Package Boundaries

@spendwise/shared enforces a clear contract: only pure validation functions cross the package boundary. The API never leaks its database types or Express internals into shared code.

No Test Coverage

There are zero test files in either package. No test runner (Jest, Vitest, mocha) is installed. Critical paths like registration, login, and validation have no automated assertions.

TypeScript Strictness Gap

No tsconfig.json is present in the repository. The code compiles via `tsx` in dev mode with default settings, so `strict`, `noUncheckedIndexedAccess`, and other safety flags are unset.

How is the code built?

Think of SpendWise like a small office building. The ground floor is the front door — it decides who gets in (security checks, rate limiting) and who to send to which department. The second floor is the authentication office where accounts are created and verified. The basement is the filing cabinet (the database) where all the user records are stored. And there's a shared rule book that everyone in the building follows so forms are always filled out correctly.

How the parts connect

When your browser sends a request, it first hits the Express server, which runs it through a chain of security checks — Helmet adds safety headers, the rate limiter prevents spam, and the request logger records timing. Then the request reaches the auth routes, which call the AuthService to do the actual work. The AuthService talks to the SQLite database to save or fetch user records, and sends the result back as a JSON response.

Index Analysis

  • idx_users_email is an explicit B-tree index on `users(email)` — accelerates the login and duplicate-check queries that filter by email.
  • The PRIMARY KEY on `users(id)` creates an implicit unique index — profile lookups by UUID are index-backed automatically.
  • No composite indexes exist yet — if expense or category tables are added, indexes on (user_id, date) and (user_id, category_id) will be needed for filtered listing queries.

Query Patterns

  • All database access uses synchronous `better-sqlite3` prepared statements — `.get()` for single-row reads, `.run()` for inserts.
  • Queries are parameterized (`?` placeholders) — SQL injection is mitigated at the driver level.
  • No transactions are used yet — the single-insert registration flow doesn't need one, but multi-table writes (expenses + categories) will require `db.transaction()`.
  • The schema migration runs inline via `db.exec()` at import time — there is no versioned migration system, so schema changes require careful manual coordination.

Backend Integration

  • database.ts creates the `Database` instance as a module-level side effect — importing the file triggers DB creation and schema migration immediately.
  • WAL mode is enabled via `db.pragma('journal_mode = WAL')`, allowing concurrent reads while a write is in progress — good for single-process performance.
  • Foreign keys are enforced via `db.pragma('foreign_keys = ON')` — ready for referential integrity when expense/category tables are added.
  • The AuthService imports `db` directly — there is no repository abstraction layer, so database logic is coupled to the controller class.
  • The DB_PATH defaults to `./data/spendwise.db` and is overridable via the `DATABASE_PATH` environment variable.

What data does the app store?

The app stores user accounts — each record has a unique ID, an email address, a securely hashed password (never the plain text), a display name, and the date the account was created. Right now there's just one table for users; as the app grows, it will add tables for expenses, categories, and budgets.

How does the app talk to the database?

When you sign up, the server takes your information, scrambles your password into a safe format, and adds a new row to the users table. When you log in, it looks up your email, checks the password, and sends back a secure token. When you view your profile, it reads your record by ID and returns your name and email — nothing sensitive like the password hash ever leaves the database.

Is the data safe?

The database uses foreign keys so that when related tables are added, deleting a user won't leave orphaned expenses behind. Every email must be unique — the database itself rejects duplicates even if the app code has a bug. A special write-ahead log mode means the database won't corrupt if the server crashes mid-write, and passwords are stored as one-way hashes that can't be reversed.

Findings

high
Hardcoded JWT Fallback Secret
packages/api/src/controllers/authService.ts:7, packages/api/src/middleware/authenticate.ts:4
Both `authService.ts` and `authenticate.ts` fall back to the string `"change-me-in-production"` when the `JWT_SECRET` environment variable is unset. An attacker who discovers this default can forge tokens for any user.
💡 RecommendationRemove the fallback entirely — throw on startup if `JWT_SECRET` is not set. Store the secret in a secrets manager or `.env` file excluded from version control. Use `crypto.randomBytes(64).toString('hex')` to generate a strong secret.
high
Wildcard CORS Fallback
packages/api/src/server.ts:27
When `ALLOWED_ORIGINS` is unset, `cors({ origin: '*' })` allows any website to make authenticated requests to the API, enabling cross-site request forgery from malicious pages.
💡 RecommendationDefault to a restrictive origin list (e.g., `http://localhost:3000` for dev). Fail closed — if no origins are configured, reject cross-origin requests rather than allowing all.
medium
In-Memory Rate Limiter Not Cluster-Safe
packages/api/src/middleware/rateLimiter.ts:5
The rate limiter stores hit counts in a plain `Map` in process memory. It resets on every server restart, doesn't share state across multiple instances, and has no eviction — the Map grows unbounded with unique IPs.
💡 RecommendationReplace with a Redis-backed rate limiter (e.g., `rate-limit-redis`) or use a proven library like `express-rate-limit` with a persistent store. Add a TTL-based eviction sweep to prevent unbounded memory growth.
medium
Unverified Client IP (Missing trust proxy)
packages/api/src/middleware/rateLimiter.ts:8
The rate limiter reads `req.ip` without `app.set('trust proxy', ...)`. Behind a reverse proxy (nginx, AWS ALB), `req.ip` returns the proxy's address — all clients share one rate-limit bucket, and attackers can bypass limits by spoofing `X-Forwarded-For`.
💡 RecommendationSet `app.set('trust proxy', 1)` (or the specific proxy hop count) so Express reads the real client IP from `X-Forwarded-For`. Validate the header against known proxy addresses.
low
Raw Error Messages Exposed in Non-Production
packages/api/src/middleware/errorHandler.ts:11
The error handler only hides raw `err.message` when `NODE_ENV === 'production'`. In staging or default environments, internal error details (stack traces, SQL errors) leak to the client.
💡 RecommendationDefault to hiding raw messages unless `NODE_ENV` is explicitly `'development'`. Consider a whitelist approach: only expose messages from errors that have a `status < 500`.

Authentication & Authorization

  • JWT tokens are signed with `HS256` (symmetric) using a secret from `process.env.JWT_SECRET` — falls back to a hardcoded string if unset, which is the top security finding.
  • Tokens carry `{ userId, role: 'member' }` claims and expire in 7 days — no refresh token rotation, so a leaked token is valid for a full week.
  • bcryptjs hashes passwords at cost factor 12 (~250 ms on modern hardware) — adequate for now, but cost should be re-evaluated as hardware improves.
  • The authenticate middleware verifies the `Authorization: Bearer <token>` header and injects `req.userId` + `req.userRole` into the Express request object via global namespace augmentation.
  • There is no role-based access control beyond setting `role: 'member'` in the JWT — no admin routes, no permission checks.

Input Validation Audit

  • Email is validated by `validateEmail` from the shared package — a simple regex that rejects whitespace and missing `@` but does NOT check DNS or MX records.
  • Password length is checked inline in `AuthService.register` (minimum 8 chars) — no complexity requirements (uppercase, digits, symbols).
  • Name is trimmed and checked for non-empty — no length upper bound, which could allow excessively long names.
  • The `express.json({ limit: '10kb' })` middleware caps request body size at 10 KB — good protection against payload flooding.
  • Category and expense validators exist in the shared package but are not yet wired into any route — they will only protect when the corresponding endpoints are built.

Dependency Risk

  • better-sqlite3 (^11.6.0) is a native addon — version pinning is important because major bumps can break the binary ABI. Currently on a caret range.
  • jsonwebtoken (^9.0.0) has had past CVEs around algorithm confusion (`none` algorithm, key/cert mismatch). The current code uses the default `HS256` which is safe as long as the secret is strong.
  • bcryptjs (^2.4.3) is a pure-JS bcrypt — slightly slower than the native `bcrypt` package but avoids native compilation issues. No known CVEs at this version.
  • helmet (^7.1.0) and cors (^2.8.5) are well-maintained with no current advisories.
  • No `npm audit` or Dependabot configuration is present — dependency vulnerabilities would go unnoticed until a manual check.

Scalability Analysis

  • The in-memory rate limiter (`Map`) resets on restart and doesn't share state across processes — a single-process bottleneck with unbounded memory growth from unique IPs.
  • SQLite is a single-writer database — it handles concurrent reads well with WAL mode but will serialize all writes, creating a bottleneck above ~100 concurrent writers.
  • The server runs as a single Node.js process — no cluster mode, no PM2, no container orchestration. A crash takes down the entire service.
  • There is no caching layer — every `/api/auth/me` request hits the database directly. Adding Redis or an in-memory LRU cache would reduce read latency.
  • The 10 KB body limit provides basic DoS protection, but there is no request-per-second limit per endpoint (the global rate limiter counts all requests equally).

Is the app safe?

ScaleHealthSickCritically IllDeath Penalty
Sick The app has solid building blocks — passwords are scrambled properly, security headers are in place, and a rate limiter stops spam — but it ships with a default secret key and wide-open cross-origin access that would let attackers forge identities if deployed as-is. It's safe for development but needs configuration fixes before going live.

What needs attention?

The biggest risk is the fallback secret key — if the server starts without a properly configured secret, anyone who knows the default can pretend to be any user. The cross-origin setting defaults to allowing every website to talk to the API, which means a malicious page could make requests on behalf of logged-in users. The rate limiter lives only in memory, so restarting the server resets all limits and running multiple servers means limits don't apply consistently.

Who can access what?

Right now, the app has two levels of access: public (anyone can register or log in) and authenticated (you need a valid token to see your profile). There's a role field in the token set to "member" but no admin role or permission system exists yet — every logged-in user has the same capabilities. The token lasts 7 days, and there's no way to revoke it early if a user's account is compromised.

Can the app handle growth?

SpendWise is built for one server and one user at a time right now. The database can handle small workloads well but would struggle with thousands of simultaneous users writing data. The rate limiter only works on a single server — add a second server and each one tracks limits independently, effectively doubling the allowed requests. To grow, the app would need a shared database, a shared rate-limit store, and the ability to run multiple copies of the server behind a load balancer.

Tell your story

We built SpendWise because personal finance apps are either too simple to trust or too complex to enjoy. Most budgeting tools either skip security entirely or bury useful features behind enterprise pricing. SpendWise starts with the hardest part done right, a secure authentication backbone with hashed passwords, token-based sessions, and layered security middleware, so every feature added on top inherits that safety by default. The monorepo architecture means a shared validation library keeps the rules consistent whether you're building a mobile app, a web dashboard, or a CLI tool. It's a foundation designed to scale from one developer's weekend project to a team-maintained product without a rewrite.

  • Bootcamp graduates building their portfolio with a project that demonstrates production-grade security practices
  • Small teams looking for a clean monorepo starter that separates API, shared logic, and future front-end packages
  • Technical interviewers who need a compact but realistic codebase for code-review exercises
  • Budget-conscious individuals who want a self-hosted expense tracker without vendor lock-in

Why does this matter?

  • Most open-source expense trackers skip authentication and security entirely, teaching bad habits from day one
  • The monorepo pattern with shared validation prevents the most common source of production bugs: inconsistent input rules across services

What makes it special?

  • Security-first foundation — Helmet, bcrypt, JWT, and rate limiting are built in from line one, not bolted on later
  • Workspace monorepo — shared validation ensures the same rules apply everywhere, from API to any future client
  • Production-aware error handling — raw errors are hidden in production, structured JSON is returned in all environments
  • Zero-dependency shared package — `@spendwise/shared` has no runtime dependencies, keeping the validation library portable and fast
  • WAL-mode SQLite — write-ahead logging gives the database crash resilience and concurrent read performance out of the box

Architecture Strengths

  • Clean middleware pipeline — each concern (security, logging, rate limiting, errors) lives in its own file with a single responsibility
  • Static service pattern — `AuthService` exposes a clear, testable API surface without instantiation overhead
  • Package boundary discipline — the shared package exports only pure functions; no Express types or database internals leak across the boundary
  • Side-effect imports for DB — `import './database.js'` ensures schema migration runs exactly once at startup, keeping the server entry point declarative
  • Centralized error shaping — the `errorHandler` middleware normalizes every error into a consistent `{ error, code }` JSON envelope

Tech Stack Justification

  • Express — the most battle-tested Node.js framework with the largest middleware ecosystem, ideal for a project that needs to be understandable by any JS developer
  • better-sqlite3 — synchronous API eliminates callback complexity, WAL mode provides concurrent reads, and zero-config file-based storage means no database server to manage
  • bcryptjs — pure JavaScript bcrypt avoids native compilation issues across platforms while providing timing-safe password verification
  • jsonwebtoken — industry-standard JWT implementation with configurable expiry, widely audited and compatible with any client library
  • Helmet — one-line middleware that sets 11+ security headers including CSP, HSTS, and X-Frame-Options, eliminating an entire category of web vulnerabilities
  • TypeScript — catches type mismatches at compile time, provides IDE autocompletion across the monorepo, and documents the API contract in code

Integration Narrative

  • The npm workspace structure means adding a `@spendwise/web` React front-end is a one-command operation — it immediately inherits the shared validation package
  • The REST API follows standard conventions (Bearer auth, JSON bodies, structured error codes) so any HTTP client — mobile, CLI, or third-party — can integrate without custom SDKs
  • SQLite's file-based storage makes the entire backend portable — copy one `.db` file to back up, migrate, or replicate the dataset
  • The ESLint shared config applies consistent code style across all packages, so new team members follow the same rules from their first commit
  • The health endpoint (`GET /health`) provides the standard interface for load balancers, container orchestrators, and uptime monitors to verify the service is alive

What if it takes off?

  • At 1,000 users, the current SQLite setup handles the load comfortably — WAL mode keeps reads fast while the single-writer model is sufficient for the request volume
  • At 10,000 users, the in-memory rate limiter becomes the first bottleneck — it can't share state if you add a second server, and memory grows with every unique IP address
  • At 50,000 users, SQLite's single-writer lock starts creating noticeable latency on concurrent expense writes — this is the trigger point for migrating to PostgreSQL
  • At 100,000+ users, the app needs horizontal scaling — multiple API containers behind a load balancer, a shared Redis cache for sessions and rate limits, and a managed PostgreSQL cluster
  • By the 250,000 user mark, you'll want read replicas for the database, a CDN for any static assets, and background job processing for tasks like email notifications and data exports

Growing pains to watch for

  • The in-memory rate limiter is the first thing that breaks at scale — adding a second server instance means limits reset independently, effectively doubling the allowed request rate
  • SQLite's file lock means only one process can write at a time — under heavy concurrent writes (many users logging expenses simultaneously), requests start queuing and response times spike
  • The lack of automated tests means every new feature or dependency upgrade risks breaking existing functionality without anyone noticing until users report bugs
  • No observability tooling (structured logs, metrics, tracing) means diagnosing production issues becomes guesswork — you can't fix what you can't measure
  • The 7-day JWT with no refresh rotation means a stolen token is valid for a full week with no way to revoke it — at scale, this becomes a real incident-response liability
